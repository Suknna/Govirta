# Govirta 项目初始化设计

## 目标

初始化 Govirta 项目仓库，为后续 MVP 和长期架构分析建立明确、可审计、可演进的基础。初始化阶段只固化项目身份、版本控制、基础文档、Go module 和技术分析文档框架；不创建业务入口、Web 框架、前端工程或 libvirt 调用代码。

## 已确认决策

- 项目名：`Govirta`。
- 远端仓库：GitHub 公开仓库 `Suknna/Govirta`。
- Go module path：`github.com/suknna/govirta`。
- 许可证：Apache-2.0。
- 远端已存在时的处理策略：远端为空仓库时复用；远端非空时失败退出，不覆盖、不强推、不合并。
- MVP 目标：实现单机 libvirt 的网页管理端。
- 长期目标：演进为面向 libvirt 的编排集群，关系类似 Kubernetes 与 Docker，并逐步替换 libvirt 当前承担的底层能力。
- 架构方向：本次方向覆盖旧的 Incus 核心运行时定位；不把 Incus 作为当前核心架构前提。
- 初始化阶段工程骨架：只创建 `go.mod`，不创建 `cmd/`、HTTP 服务入口、daemon 入口或前端目录。
- 技术分析范围：同时覆盖 MVP 与长期演进路径。

## 推荐方案

采用“文档优先的显式初始化”。该方案优先落盘项目边界、协作规则和技术分析框架，避免在缺少技术分析前过早固定 API、Web、daemon 或节点代理的入口语义。

## 仓库与版本控制设计

初始化操作应创建本地 git 仓库并使用 `main` 作为默认分支。远端仓库应指向 GitHub 公开仓库 `Suknna/Govirta`。

远端处理规则必须 fail-closed：

1. 如果远端仓库不存在，创建公开仓库。
2. 如果远端仓库存在且为空，复用该远端。
3. 如果远端仓库存在且非空，停止初始化并报告原因。
4. 禁止强推、覆盖、自动合并或静默迁移远端内容。

## 初始化文件设计

初始化完成后应包含以下文件：

- `README.md`：说明项目定位、MVP 范围、长期方向、当前阶段非目标和开发状态。
- `AGENTS.md`：为后续 agent 明确项目目标、阶段边界、显式优先原则、技术分析优先级和禁止事项。
- `.gitignore`：覆盖 Go、编辑器、本地临时目录和构建产物；保留源码、文档和配置文件。
- `LICENSE`：Apache-2.0 标准许可证文本。
- `go.mod`：声明 `module github.com/suknna/govirta`。
- `docs/technical-analysis.md`：建立 MVP 与长期演进的技术分析框架，记录需要验证的官方文档和开源项目方向。

初始化阶段不创建以下内容：

- `cmd/govirta/main.go` 或任何可运行服务入口。
- Web 框架、前端工程、路由层或 UI 组件。
- libvirt Go binding 的具体调用代码。
- 数据库 schema、迁移目录或运行时配置系统。
- 集群控制面、节点 agent、调度器或控制循环实现。

## README 内容设计

`README.md` 应清晰说明：

- Govirta 是一个以 Go 为主要语言的虚拟化管理项目。
- MVP 聚焦单机 libvirt Web 管理端。
- 长期方向是演进为面向 libvirt 的编排集群，并逐步替换 libvirt 当前承担的底层能力。
- 当前初始化阶段不提供可运行服务。
- 后续技术选型必须基于官方文档、活跃开源项目和可复现实验，而不是训练记忆或隐式默认。

## AGENTS.md 内容设计

`AGENTS.md` 应作为项目内 agent 协作准则，至少包含：

- 项目阶段：初始化期、MVP 期、长期编排期。
- 核心产品方向：单机 libvirt Web 管理端起步，长期演进为编排集群。
- 显式优先原则：禁止隐藏默认、魔法推断、静默 fallback、兼容别名和自动翻译行为。
- 技术分析优先：引入第三方库、SDK、框架或关键架构前必须查阅官方文档并评估开源项目活跃度。
- 简化优先：初始化阶段不引入业务骨架；后续实现也应避免过早抽象。
- 验证要求：所有完成声明必须附带命令或可复现证据。
- 文档约束：架构决策、非目标、迁移边界和运行时语义必须显式写入文档。

## 技术分析文档设计

`docs/technical-analysis.md` 初版应建立分析框架，而不是伪装成已经完成最终选型。文档应覆盖：

- MVP 问题域：单机 libvirt 管理、Web/API 边界、本地执行权限、状态读取与操作下发。
- 长期问题域：声明式资源模型、控制循环、节点代理、调度、状态存储、网络与存储抽象。
- 参考模型：Kubernetes 的声明式 API、控制器、kubelet/节点代理、资源对象生命周期；libvirt 的管理边界；现有虚拟化管理项目的产品边界。
- 后续调研清单：libvirt 官方文档、Go libvirt binding、Cockpit、Kimchi、oVirt、Proxmox VE、Harvester、OpenNebula 等。
- 明确未决问题：是否直接操作 libvirt、是否引入本地 agent、Web 管理端与 API server 如何分层、长期替换 libvirt 能力的迁移顺序。

## 验证设计

初始化实现完成后必须执行以下验证：

1. `git status --short --branch`：确认当前分支和工作区状态。
2. `go test ./...`：确认 Go module 基础状态可被 Go 工具链识别。
3. `go list ./...`：确认 module 包列表行为符合初始化阶段预期。
4. `git remote -v`：确认远端 URL 指向 `Suknna/Govirta`。
5. GitHub 远端检查：确认公开仓库存在，且未覆盖非空远端。
6. 提交检查：确认初始提交只包含初始化目标相关文件。

## 风险与约束

- 如果 GitHub CLI 未登录或无权创建 `Suknna/Govirta`，应停止远端创建并报告实际错误，不得改用其他账号或仓库名。
- 如果远端仓库已存在且非空，应停止初始化，不得自动拉取、合并或覆盖。
- 如果 `go test ./...` 因当前没有包而返回非零，应记录 Go 工具链的实际输出，并在后续实现计划中决定是否通过创建空包或调整验证命令解决；初始化阶段不因此创建业务入口。
- 技术分析中涉及第三方库或 SDK 时，必须先查官方文档；无法查询时要显式标注降级原因。

## 成功标准

项目初始化任务完成时，应满足：

- 本地仓库存在并使用 `main` 分支。
- GitHub 公开远端 `Suknna/Govirta` 存在并与本地仓库关联。
- 基础文件已创建，内容与本设计一致。
- Go module path 已固定为 `github.com/suknna/govirta`。
- 不存在业务入口或过早引入的框架代码。
- 技术分析文档已建立 MVP 与长期演进的分析框架。
- 所有完成声明均附带验证命令与结果。
